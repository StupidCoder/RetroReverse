# --- the sprite/level-bank loader and codec (Part IV §3 tile map, §4 obstacles) ---
0080B4  ilb_load         load an .ilb/.vlb sprite bank for the current course: index a buffer table, read+unpack the WHOLE file (bank_read_unpack — one ByteRun1 stream), then walk the bank's descriptors IN the unpacked buffer at buf+2, stride $14=20. Each 20-byte descriptor: +0 type byte (1=stored cell, 0=composited), +1 flag byte (bit0 = process), +2 width in 16-px words, +4 height in rows, +6 one-plane byte size (=w*2*h), +8 source offset (relocated +buf), +$C dest cell ptr (allocated from buffer slack or relocated), +$10 aux ptr (relocated). Cells' source pixel data is contiguous after the descriptor array; plane count = source span / +6 (2 for small objects, 4 for scenery). Composites type-0 cells via composite_planes.
007F38  mlb_load         load an .mlb level module (different container: flag $01 + a 5-entry offset-table header) through the SAME bank_read_unpack path, so it shares the ByteRun1 codec.
007E4E  bank_read_unpack file loader: dos_open(name, MODE_OLDFILE) -> dos_read into the buffer -> dos_close; then, if the global flag $8351 is set, move the loaded (packed) data to the high end of the buffer and ByteRun1-decompress it in place (unpack_byterun1) back to the start. Returns the unpacked length.
007DA6  pool_alloc       bump allocator: hand out $8(a6) bytes from the work pool tracked by $8340/$8344 (high-water at $8334), growing it via $8D74 when exhausted.
009118  unpack_byterun1  THE BITMAP CODEC = ByteRun1 / PackBits (identical to IFF ILBM bodies). unpack(src@8, dst@C, srclen@10, dstlen@14): read a signed control byte n — n in 0..127 copies the next n+1 bytes literally; n in -1..-127 repeats the next single byte (1-n) times (2..128); n = -128 is a no-op. Stops at dstlen or end of source.
008026  composite_planes one bank cell: zero the dest cell (size = desc->4 * desc->2 * 2), then OR four consecutive planar source blocks (desc->8) into it — builds the final 2-plane (16-wide) cell from the unpacked source.

# --- top level: the game shell and the real-time "Framer" main loop ---
# main ($243E8, the C entry's argv setup at cstart) calls game ($2B68).
002B68  game_shell       opens intuition.library, runs init/draw (sub_00CA14, which reaches the render chain) and start_gameplay ($6FC6), then an IDCMP message loop: GetMsg ($249B4) / ReplyMsg ($249C8), dispatch on the message Class field (+$14 == 8/9/10) to the start-course handlers ($3410/$3432) or quit. This is the menu/window front-end, NOT the per-frame loop.
006FC6  start_gameplay   thin wrapper over $6F20(flag); $6FDC creates the real-time task.
006FDC  create_framer    create an exec task named "Framer" (name string at $70D8) via $248D4, entry = framer_task $6DA8. Wires up its frame-rate-tracking globals ($7060/$7092/$7096 handler ptrs, $70B8 frame-time histogram).
006DA8  framer_task      THE real-time main loop (own task). Each iteration: Wait ($24B54) for the vblank signal, then call frame_tick $7596(frame_index); meanwhile it tallies how many vblanks each frame took into the $70B8 histogram (the "Framer" = frame-rate meter) and polls GetMsg ($249B4) for a quit/stop message to exit.
007596  frame_tick       one displayed frame of work (the Framer task's per-vblank handler): runs the two colour-cycling passes ($B6DE, $A7A2), per-subsystem gates ($5BCA, $1E20D), and on a fresh world frame ($5DB) advances the frame counter $230 and latches the scroll position ($D30C -> $7692); if anything changed, rebuilds the display ($86B4).
00B6DE  colour_cycle_a   advance one set of colour-cycling effects: walk the $236 array (stride $F4), step each entry's $7B counter (wrap $20), look up the next $0RGB from the cycle table ($B774->copper $B954=COLOR11 / $B794->copper $B960=COLOR13) and write the copper colour slot. Returns 1 if any changed. $B774 = 0F00..0FFF (RED<->WHITE, =$A86E), $B794 = 000F..0FFF (BLUE<->WHITE, =$A88E). So COLOR11/12 are the red/hazard pair, COLOR13/14 the blue/ice pair.
00A7A2  colour_cycle_b   a second colour-cycle pass over the $236 array using the $7A counter (wrap $40, index = counter>>2 = 16 steps) and tables $A86E/$A88E -> copper slots $B95A/$B966. Each table = 16 $0RGB words: $A86E = 0F00,0F22..0FFF..0F00 (RED<->WHITE hazard, copper $B95A = COLOR12) ; $A88E = 000F,022F..0CCF,0FFF..000F (BLUE<->WHITE = the ICE shimmer, copper $B966 = COLOR14). The $19(a5) per-object flag picks blue(ice) vs red(hazard). The ICE PIT (beginr) uses BOTH blue slots COLOR13 (.mlb CCF) + COLOR14 (.mlb 000). (Part IV §3: extract/cmd/sprites fills black .mlb cycle slots with a MID-cycle frame -- COLOR14 -> $088F cyan, not the pure-blue endpoint $000F -- so the ice renders as cyan shades.)
0086B4  rebuild_display  rebuild the per-frame copper/display list from the Track display data ($89C2) and the latched scroll position ($7692); appends nodes via the list-builder $8688.
00CA14  game_state       the gameplay STATE MACHINE (dispatched on $CDAC): 1 = set up the course (reset scroll $D30C, DMACON, counters), 2 = ..., 3 = play (the per-object world update + render, reaching object_draw $014EF0 via $00CD90). Called from game_shell; the per-world-frame tick mechanism (vs the Framer's vblank) is the open link.

014EF0  object_draw       per-object update+draw: advance the object's world pos by its velocity (obj+$C += obj+0, +$10 += +4, +$14 += +8), set_draw_pos, then draw the object plus up to three actor passes (a main sprite + shadow/secondary passes) chosen by the object subtype obj+$1B (==1/2/3), each at its own Y offset ($1E8/$238/$210) via actor_update; a per-frame "already drawn" bitmask ($5F6) prevents double-draws.

# --- marble physics & controls (Part V §4) ---
# PROVEN velocity facts: the marble (player 1) is object $236 (player 2 $32A); its
# velocity is obj+0/+4/+8 and position obj+$C/+$10/+$14 (object_draw $14EF0 at
# $14FFC integrates pos += velocity).
013E9A  marble_state_machine  THE marble STATE MACHINE (a5=marble). d7 = obj+$1A = the
#   marble state, 0..11 (12 states). Prologue $13EAC: for most states runs a wobble/settle
#   anim (obj+$EA gate, obj+$72/$73 bob counter+dir, obj+$7A timer); states 2,4,7,A,B skip
#   it. $13F2C: d0=state; CMPI.l #$C; BCC $14486 (default = state>=12); ASL #2; JMP
#   $2(pc,d0.l) -> 12-entry table at $13F40. obj+$1A doubles as the per-surface max-speed
#   selector read by $14E7E ($14D28). The 12 handlers (●=player-controllable: input $12F8C
#   + physics $14EF0; others redraw at current pos via $E74C, no input):
#     s0 $141DC ● ROLLING (normal play); if stun flag +$DD set -> state 8.
#     s1 $1427C ● LANDING: marble drops off a ledge -> airborne (+$36=2, in the update
#                 $14EF0); on touchdown $14884 sets s1 with the fall height -> impact-scaled
#                 bounce anim (+$60 eased to +$61), -> 0.
#     s2 $1436E   animation frame only (advances anim cursor +$64).
#     s3 $13FBA   EDGE/LANDING reaction: plays the sound for terrain +$62, clears, -> 0
#                 (entered by the wall/edge handlers $16xx/$17xx).
#     s4 $14072   FALLING/SETTLING: collision-query $15FC8; on contact runs appear-anim +
#                 latch region $14706, -> 0.
#     s5 $142FC ● OBJECT BUMP: the collision scan $1A86C (Track actor list $1ABE0, only
#                 courses $5D6 in {1,2,5}) drops the marble here on contact -> flat +$60
#                 countdown, -> 0.
#     s6 $13F70 ● COURSE INTRO entrance run (entered at course_init $3380); +$61 down, freeze vel.
#     s7 $1435E   transient: save-state + clear +$1C.
#     s8 $143FC   DIZZY (stunned): plays the swirl anim +$DE, tallies +$74, -> 0. Entered
#                 ONLY from s0 on a hard hit / survivable fall. RECOVERABLE, NOT death.
#     s9 $1439A   SPAWN/appear (entered from the spawn routine $14804).
#     s10 $14100  HOLE/REGION CAPTURE: runs the active region's script $FD68 + sound, -> s4
#                 (entered from the slope handlers $17AFA/$17CDA).
#     s11 $143C8  settle onto a scripted region (like s4, latches on contact).
# THE STATIC-SLOPE FORCE (the long-open path: how the $9A6 height MESH rolls the marble):
#   NOT sub_016900 (that is the scripted dynamic regions). It is the HEIGHT-FIELD GRADIENT,
#   read each frame by object_draw -> $EA10 then applied by $14A88.
00EA10  surface_sample   read the marble's 2x2 mesh cells (the four 8-byte corner-height
#   blocks $1EBA/$1EC2/$1ECA/$1ED2, filled from the $CCA mesh) and interpolate. $6D6 (iso
#   diamond half) picks which TRIANGLE of the cell the marble is in; $6D8/$6DA = the surface
#   GRADIENT (corner-height differences along x/y); then height = $6D8*$6D2 + $6DA*$6D4
#   (sub-cell), <<$D, + cornerZ<<$10 = the interpolated surface Z. So the triangular slope
#   faces are real: the gradient is taken from one of the two triangles of the tile quad.
00EF90  surface_sides    classify the 4 sides of the marble's cell from neighbour corner
#   heights ($1EBA..$1ED2): set side TYPE flags $6A4-$6A7 (0/1/3 = none/edge/wall) and side
#   values $6A8/$6AA/$6AC/$6AE. Feeds edge collision $EB64.
00EB64  edge_collision   per-side wall/edge response: when the marble's sub-cell pos
#   ($6D2/$6D4) crosses a boundary and that side's flag ($6A4-$6A7) is a wall, clamp the
#   velocity component ($EB2C/$EB48 adjust obj+0/+4).
014A88  apply_slope_force  THE static-slope downhill force (a5=marble). Read gradient
#   $6D8/$6DA (from surface_sample $EA10), x4 if steep (|g|>=$C), then marble velocity
#   -= gradient<<$B:  $14AEA SUB.l d0,(a5) vX -= gX<<11 ; $14AF6 SUB.l d0,$4(a5) vY -= gY<<11
#   = accelerate DOWNHILL (down the height gradient). Course 4 (aerial $5D6==4) ADDs instead
#   = inverted gravity. Called from the rolling-state handler at $0156DC.
012F8C  marble_input     INPUT -> velocity. For the marble ($236) reads the trackball
#   force ($195EA/$195EE; player 2 $195EC/$195F0), NEGates X, scales (*$160, >>8, * a
#   per-state factor from obj+$60/$1A, <<3), and ADDS it to the velocity:
#   $0130DA ADD.l d0,(a0)  -> obj+0 (vY) += force ;  $0130E0 ADD.l d0,$4(a0) -> obj+4
#   (vX) += force. (Then JSRs $14E7E.) PROVEN, no assumptions.
00DEBC  build_surface    $9A6 course descriptor -> $CCA surface-region structs. Reads
#   the descriptor (a0=$9A6): +$08 command/coordinate stream (bit selector + [col,row]
#   +value words, $FFFE/$FFFF terminated), +$18 world height, +$1A region count, +$1C
#   region-record table, +$20 edge-run sub-shape ptr table, +$24 width, +$26 per-column
#   word array (->$9AE/$9B2). Clears the $CCA buffer ($420 longs), then for each region
#   calls sub_00E158 to rasterise an 8-byte region record into the $CCA structs. PROVEN:
#   $9A6 terrain -> $CCA region structs.
00E158  build_region     rasterise one 8-byte region record into the $CCA CORNER-HEIGHT
#   mesh (a 22-wide grid of 8-byte cells, each = 4 corner heights at +0/+2/+4/+6). Record
#   = [+0 origin.x(s8)][+1 origin.y][+2 size.x][+3 size.y][+4..5 baseHeight word][+6 low5 =
#   edge-shape selector into the $9A6+$20 table -> a height-PROFILE byte array][+7 low3 =
#   SLOPE DIRECTION 0..7, indexes $2504 for the step (dx,dy); bit3 = flip]. $2504 holds
#   only the 4 iso diagonals (+/-1,+/-1); dir picks 1 of 8 fills (4 diagonals x 2 loop-
#   transpose orders, dir<4 vs dir>=4 take the two inner loops $E274/$E464). Walks the
#   rectangle DIAGONALLY in iso space: per step cell row=xAcc+yAcc-originX, col=originY-
#   xAcc+(row>>1), addr=$CCA+(row*$16+col)*8 (odd rows clamp to 21 cols = iso half-cell).
#   Writes value = baseHeight +/- profile[i] (flip = sign), profile advanced 1 byte/cell
#   and RESTARTED on a $80 marker (so 00 02 04..80 = repeating linear ramp) -> the height
#   RAMPS along the diagonal = the slope. A boundary test writes value into the right
#   corner word(s) of the cell depending on left/right (x0/xEnd) and top/bottom (y0/yEnd)
#   edges, sharing corners so adjacent cells agree = a continuous corner-height terrain.
#   NOTE $CCA is a MULTI-PHASE work buffer (verified same literal $0CCA at 3 strides):
#   build_region writes 8-byte corner-height cells; region_clear $8F42 inits 86-byte
#   region structs; particle/effect system $1E21C reuses 8-byte pos+vel records.
008F42  region_clear     init all 25 $CCA region structs (stride $56): +$18=0 (invalid),
#   +$C=0 / +$10=0 (LONG ref point), +$1F=0 (terrain code), +$19=index(0..24). So the ref
#   point starts at ZERO and is filled later by region_script $FD68 (NOT a static field).
00F7EA  region_build_list  populate the 86-byte $CCA region structs from a per-course
#   source list (head = *($FD2C)/Track header+0). Walks [x][y][scriptPtr-long] records; per
#   entry (filtered by tile-range args) allocs a struct via $F6B2, sets +$52/+$54 = (x,y)
#   grid key, and calls region_update $F96A which stores the record's long as the region's
#   SCRIPT POINTER +$3A and sets +$18/+$1A. region_activate $F8FC matches +$52/+$54 vs the
#   marble tile each frame to set +$18 valid. NB the record [x][y] (+$52/+$54) is only a
#   TRIGGER/grid-activation CELL (a single-cell equality/bracket match in $F8FC, NOT a w*h
#   rect) — like a creature's home cell. A dynamic region has a POSITION, no SIZE: its actual
#   world position is the script's first op0 KEYFRAME refX,refY (-> +$C/+$10 = refX<<19), and
#   spatial extent (e.g. the whole drawbridge) is built by TILING several point-regions.
#   (Contrast the STATIC $9A6 slope records, which DO carry [xSize][ySize].)
00FD68  region_script    PER-REGION BYTECODE INTERPRETER (a5=region) — THIS writes the
#   downhill reference +$C/+$10. +$3A=script base, +$36=script PC (advances 2B/word read),
#   +$1E=current keyframe DURATION in frames (counted down; at 1 fetch next opcode). Word
#   opcode 0..$12 -> 19-entry jump table $FD96 (target=entry+2+off). KEY OPS:
#   op0 KEYFRAME ($FDE2): read 5 words -> +$C=refX(word<<$13/19), +$10=refY(word<<19),
#     +$14=refZ(word<<$10/16), +$1E=duration, +$1F=terrain code(5..59). Plants the ref pt.
#   op16 MOVE ($1002E): each frame +$C += region.vX(+0), +$10 += region.vY(+4) -> the ref
#     point DRIFTS = a MOVING slope/seesaw/sliding wall.
#   op2 ($1007A): set +$1C(sprite/state word)/+$23/+$22, restore +$3E=+$46; if +$1F==$A sound.
#   So a scripted region's +$C/+$10 = a KEYFRAME STREAM (from the Track anim-script table
#   $FD2C). Fixed-point <<19 matches the marble position scale so sub_016900 subtracts directly.
# THE 19-OPCODE JUMP TABLE $FD96 (op*4; target=$FD96+2+off). ALL decoded (a5=region, PC=+$36);
# the interpreter runs ops in a loop (flag -$D) until a STOP op clears it. Reproduced by the
# tool extract/cmd/rgnscript (disassembles any course's region scripts). Operand bytes shown:
#   op0  $FDE2  KEYFRAME  5w: refX,refY,refZ,dur,terr -> +$C=refX<<19/+$10/+$14, +$1E dur,
#     +$1F terr.  IF dur==1 also reads +$26,+$28 (2w) and a LINK long -> +$46/+$3E (a sprite-
#     anim ptr). Plus terrain self-REGISTRATION: terr3->$FD38 (THE drawbridge), terr12->$FD58
#     funnel list, terr11/13->$FD3C slope list, terr10->$FD54, terr6/25 special. (not a STOP)
#   op1  $FFF0  STATE-STOP   2w(+$1C,+$21): state +$1A=1; STOP.
#   op2  $1007A SPRITE       2w(+$1C,+$23): set sprite/state, base=+$46. (not a stop)
#   op3  $100D2 STATE0-STOP  1w(+$1C): state 0; STOP.
#   op4  $101A0 LOOP-A       1w(count->+$24); mark loop top +$2A.
#   op5  $101C2 NEXT-A       0w: dec +$24, branch to +$2A while !=0 (count 0 = infinite loop).
#   op6  $101EC LOOP-B       1w(count->+$25); top +$2E.    op7 $1020E NEXT-B (2nd dbra).
#   op8  $100EC IF-MARBLE-ON-REGION 1w arg + LONG target: if a marble (obj $236+i*$F4) has
#     state +$18==1 and is within 3 tiles of the region ref, JUMP target (else skip). NB this
#     is NOT what the drawbridge uses (earlier mis-parse). practy region 0 uses it = the
#     AUTO-START RAMP (detects the idle marble at the start, grows a ramp to nudge it).
#   *** CONTROL FLOW (how scripts chain): ***
#   op9  $10238 JUMP  LONG : PC = target.
#   op10 $10244 CALL  LONG : PC = target, save return -> +$32 (gosub).   op11 $1025C RETURN.
#   op12 $10266 LINK-46 LONG: +$46/+$3E = ptr (load a sprite-anim/sub-script, continue inline).
#   op13 $10286 LINK-4A LONG: +$4A/+$3E = ptr.
#   op14 $102A6 SET-VEL 2w  : region vX(a5+0)/vY(+4) (feeds op16; <<8 when dur==1).
#   op15 $102E6 ACTIVATE-STOP 0w: JSR region_update $F96A(region,1,0); STOP.
#   op16 $1002E MOVE 0w     : when +$1E==1, drift +$4E/+$50 by vX/vY = a moving slope/wall.
#   op17 $10300 FALL-STOP 1w(+$1B): marble-capture state +$1A=4 + code +$1B, save PC; STOP.
#   op18 $10326 IF-MARBLE-TERRAIN LONG: if marble terrain +$1B matches region +$1B(range>=$1E)
#     JUMP target + JSR $21CD0(table $20A4[code-$1E]).
# THE DRAWBRIDGE (beginr region 0, script $1C1E) decompiled by rgnscript: op0 KEYFRAME(61,56)
#   terr3 (registers as $FD38) -> op4 LOOP-A 0 (forever) { op2 SPRITE; op12 LINK $1A8A; op3
#   STOP +$1C=15 ; op2 SPRITE; op12 LINK $1A4E; op3 STOP +$1C=30 } op5 NEXT-A. = an AUTONOMOUS
#   timed open/close cycle alternating two linked bridge animations ($1A8A/$1A4E), holding 15
#   then 30 ticks. NOT marble-triggered. (Marble_Madness.md Part IV.5 has the annotated listing.)
# FUNNELS (beginr regions 3-7): each script is just op0 KEYFRAME(terr) + op3 STOP. The
#   entrance->exit LINK is NOT in the Track data - it is HARDCODED in the surface_interaction
